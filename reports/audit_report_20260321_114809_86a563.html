A.8.2 — Privileged Access Rights

FAIL SSH Root Login Disabled A.8.2.1

Direct root login via SSH must be disabled to enforce accountability and prevent brute-force attacks on the root account.

Severity: HIGH
Expected: no
Actual:
⚠ Remediation Steps
1. Edit SSH configuration:
   sudo nano /etc/ssh/sshd_config
2. Set or add the following line:
   PermitRootLogin no
3. Restart the SSH service:
   sudo systemctl restart sshd
4. Ensure you have a non-root user with sudo access before making this change.
FAIL No NOPASSWD Sudo Entries A.8.2.2

Sudo access should require password authentication. NOPASSWD entries allow privilege escalation without verification.

Severity: HIGH
Expected: false
Actual: True
⚠ Remediation Steps
1. Edit the sudoers file safely:
   sudo visudo
2. Find and remove or modify any lines containing NOPASSWD:
   user ALL=(ALL) NOPASSWD: ALL  →  user ALL=(ALL) ALL
3. Also check files in /etc/sudoers.d/:
   sudo ls -la /etc/sudoers.d/
   sudo visudo -f /etc/sudoers.d/
4. Remove the NOPASSWD tag from each entry.
PASS Sudoers File Permissions A.8.2.3

The /etc/sudoers file should have restricted permissions (0440) to prevent unauthorized modifications.

Severity: MEDIUM
Expected: 0440
Actual: 0440

A.8.5 — Secure Authentication

FAIL SSH Key-Based Authentication Only A.8.5.1

Password-based SSH authentication should be disabled in favor of key-based authentication to prevent brute-force attacks.

Severity: HIGH
Expected: no
Actual:
⚠ Remediation Steps
1. First, ensure SSH key-based authentication is set up:
   ssh-keygen -t ed25519 -C "your_email@example.com"
   ssh-copy-id user@server
2. Test key-based login works before disabling passwords.
3. Edit SSH configuration:
   sudo nano /etc/ssh/sshd_config
4. Set:
   PasswordAuthentication no
   PubkeyAuthentication yes
5. Restart SSH:
   sudo systemctl restart sshd

Severity: HIGH

4. Ensure PAM is configured to use pwquality:
   sudo nano /etc/pam.d/common-password
   Add/modify: password requisite pam_pwquality.so retry=3

Severity: HIGH
Expected: true
Actual: False
⚠ Remediation Steps
1. Install ClamAV:
   sudo apt update
   sudo apt install clamav clamav-daemon
2. Update virus definitions:
   sudo systemctl stop clamav-freshclam
   sudo freshclam
   sudo systemctl start clamav-freshclam
3. Enable and start the daemon:
   sudo systemctl enable clamav-daemon
   sudo systemctl start clamav-daemon
4. Configure scheduled scans:
   sudo crontab -e
   Add: 0 2 * * * /usr/bin/clamscan -r /home --log=/var/log/clamav/daily_scan.log

Audit rules should be defined to monitor critical security events such as file access, user authentication, and privilege changes.

Severity: MEDIUM
Expected: 0
Actual: 0
⚠ Remediation Steps
1. Create audit rules file:
   sudo nano /etc/audit/rules.d/audit.rules
2. Add recommended rules:
   # Monitor login/logout events
   -w /var/log/faillog -p wa -k logins
   -w /var/log/lastlog -p wa -k logins

   # Monitor user/group changes
   -w /etc/passwd -p wa -k identity
   -w /etc/group -p wa -k identity
   -w /etc/shadow -p wa -k identity

   -w /etc/sudoers -p wa -k sudo_changes
   -w /etc/sudoers.d/ -p wa -k sudo_changes

   # Monitor SSH config
   -w /etc/ssh/sshd_config -p wa -k sshd_config

   # Monitor system calls for privilege escalation

A.8.22 — Segregation of Networks

FAIL IPv4 Forwarding Disabled A.8.22.1

IP forwarding should be disabled unless the system is acting as a router, to prevent it from being used to bypass network segmentation.

Severity: MEDIUM
Expected: 0
Actual: 1
⚠ Remediation Steps
1. Disable IP forwarding immediately:
   sudo sysctl -w net.ipv4.ip_forward=0
2. Make the change persistent:
   sudo nano /etc/sysctl.conf
   Add or modify: net.ipv4.ip_forward = 0
3. Apply:
   sudo sysctl -p
4. Verify:
   cat /proc/sys/net/ipv4/ip_forward
   Should output: 0

Severity: MEDIUM
Expected: 0
Actual: 0

Severity: MEDIUM
Expected: 0
Actual: 1
